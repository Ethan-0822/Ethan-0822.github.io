入门解析#

⭐parent【管理依赖】#

✅模板创建的项目的pom#

  • springboot模板中配置一整套依赖和插件，这些依赖和插件是官方测试过的兼容好的插件，只需要继承spring-boot-starter-parent.pom就能获取到这些依赖和插件了

    • <parent>
   <groupId>org.springframework.boot</groupId>
   <artifactId>spring-boot-starter-parent</artifactId>
   <version>2.6.1</version>
   <relativePath/>
</parent>
✅spring-boot-starter-parent.pom#

  • 开发者在进行开发时，往往对依赖版本的选择具有固定的搭配格式，如A技术的2.0版与B技术的3.5版可以兼容，但是和B技术的3.7版合并使用时就有冲突。于是SpringBoot就将所有的技术版本的常见使用方案整合了，如A，B技术的固定搭配，开发者只需要使用springboot提供的方案【即继承spring-boot-starter-parent】，就能安心使用各种依赖，不用再担心冲突问题。

  • 【parent定义出来以后，并不是直接使用的，仅仅给了开发者一个说明书，但是并没有实际使用】 进入spring-boot-starter-paren.pom文件之后，它看到存在于本地仓库org\springframework\boot\spring-boot-starter-parent\2.6.1中，最主要的是spring-boot-starter-paren.pom也继承一个spring-boot-dependencies.pom

    • <parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-dependencies</artifactId>
  <version>2.6.1</version>
</parent>
✅spring-boot-dependencies.pom#
  • 这个pom文件中中定义了两组信息
    • 第一组是定义了非常多依赖版本号属性，属性值指定了若干个技术的依赖版本号
    • 第二组是配置了非常多依赖坐标，这些依赖坐标中没有具体的依赖版本号，而是引用了第一组信息中定义的版本号属性
  • 所以springboot在该pom文件统一管理了所有的资源，而且是稳定性与兼容性已经测试过的，甚至于已经配好了固定的版本，直接继承和引用依赖就可以了，都不需要指定版本
  • 注意：
    • 使用idea连接阿里云创建springboot模板，是直接继承spring-boot-dependencies.pom的
    • 【不同的springboot版本，具体的依赖和插件可能有差别】

⭐starter【导入与配置依赖】#

✅创建项目时选取技术类型的依赖#

  • parent总依赖中定义了若干个依赖版本管理，但是parent只是管理依赖【未导入】，starter是导入和配置需要的依赖

  • 选取技术类型后，比如选中了web，可以看到创建完项目后，pom文件已经导入了spring-boot-starter-web，starter就会自动将web技术所有的依赖与配置项全部处理，直接开箱即用

  • <!--选取web技术类型-->
<dependencies>
   <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-web</artifactId>
   </dependency>
</dependencies>
✅spring-boot-starter-web.pom#

  • 在右侧的maven依赖中，可以看到spring-boot-starter-web依赖已经配置好了spring-webmvc，spring-web，spring-boot-starter-tomcat等等的依赖。所以通过技术选型选取依赖，通过依赖传递，将选取的依赖传递进新创建的项目

  • 【每个starter代表不同的功能，根据不同的技术选型，选取对应的starter后就可以实现快速配置的效果】

  • 【实际开发中如果需要用什么技术，先去找有没有这个技术对应的starter】

    • 如果有对应的starter，直接写starter，而且无需指定版本，版本由parent提供
    • 如果没有对应的starter，手写坐标即可

  • 【实际开发中如果发现坐标出现了冲突现象，确认你要使用的可行的版本号，使用手工书写的方式添加对应依赖，覆盖SpringBoot提供给我们的配置管理】

    • 方式一：直接写坐标
    • 方式二：覆盖中定义的版本号，就是下面这堆东西了，哪个冲突了覆盖哪个就OK了

⭐引导类【运行】#

  • 目前程序运行的入口就是SpringBoot工程创建时自带的一个类了，这个类带有main方法与@SpringBootApplication注解，运行这个类就可以启动SpringBoot工程的运行

    • @SpringBootApplication
public class Application {
    public static void main(String[] args) {
        //启动spring容器
        ConfigurableApplicationContext context = SpringApplication.run(Application.class);
    }
}

  • SpringBoot本身是为了加速Spring程序的开发的，而Spring程序运行的基础是需要创建自己的Spring容器对象（IoC容器）并将所有的对象交给Spring的容器管理，当前这个类运行后就会产生一个Spring容器对象，并且可以将这个对象保存起来，通过容器对象直接操作Bean。

  • 观看@SpringBootApplication注解源码

    • 可以看到中包含了@Configuration，而@Configuration用于指定当前类是一个spring配置类，当容器创建时会从该类上加载注解
    • 也可以看到包含了@ComponentScan，而@ComponentScan用于指定在spring初始化容器时需要扫描的包【作用与spring的xml配置文件一样，只是从xml方式变成注解的方式】

  • 【注意：没有指定扫描包的情况下，引导类会默认扫描当前包以及子包】

⭐内嵌tomcat【调试】#

  • 由于导入了starter-web依赖，而starter-web又内嵌的tomcat依赖

  • 同时tomcat也是使用Java语言开发的，所以springboot将tomcat核心抽取出来作为一个对象，放入spring容器中。spring启动时，就启动了tomcat对象【对，tomcat服务器运行其实是以对象的形式在Spring容器中运行的】

  • springboot一共支持3种web服务器web服务器

    • tomcat(默认)：apache出品，粉丝多，应用面广，负载了若干较重的组件
    • jetty：更轻量级，负载性能远不及tomcat
    • undertow：负载性能勉强跑赢tomcat
    • 在org.springframework.boot.autoconfigure.web.servlet.ServletWebServerFactoryAutoConfiguration【Web服务器自动配置类】可以看到

  • 使用starter类型的依赖就能导入需要web服务器依赖，如：

    • 【springboo默认内嵌的tomcat依赖，所以需要在-starter-web排除tomcat依赖】

    • <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <!--排除默认内嵌的tomcat依赖-->
    <exclusions>
        <exclusion>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-tomcat</artifactId>
        </exclusion>
    </exclusions>
</dependency>

<!--导入jetty服务器依赖-->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-jetty</artifactId>
</dependency>

⭐配置文件格式分类#

  • SpringBoot提供了3种配置文件的格式

    • properties【传统格式/默认格式】
    • yml【主流格式】
    • yaml

  • SpringBoot配置文件加载顺序

    • 场景：当其中两种文件格式，或者其中三种文件格式，存在某个相同的配置项，那么该配置项的加载优先级如下

    • application.properties  >  application.yml  >  application.yaml

    • 如果没有包含同一个配置项，那么即使是不同类型的配置文件也会将所有的配置项全部保留下来

  • 表示注释
  • 可以代表字面值、对象数据格式、数组数据格式

public class TbUserController {

    private TbUserService tbUserService;

    /**
     * 由于类已经添加了@RequestMapping("/users")，且此处@GetMapping没有参数
     * 当使用get方式访问/users【相当于访问这个类的默认方式】时，如http://localhost:80/users/
     * 就会调用此函数

    @GetMapping

     * @param id
     * http://localhost:80/users/4

     * @return
     */
    @GetMapping("{id}")
    public TbUser getById(@PathVariable Integer id){
        return tbUserService.getById(id);
    }

}

热部署#

✅场景#
  • 正常情况下，想要代码发生变化后生效，需要重启springboot程序，这样idea就会重新编译代码，加载进springboot集成的tomcat容器中。但是遇到大型的程序，重启项目程序需要的时间可能会非常长，这个时候可以使用热部署来加载新的代码。
✅原理#
  • 本地开发时，如果使用tomcat容器来热部署的话，tomcat是感受不到springboot程序发生变化的，因为本地的tomcat服务器是受springboot管控的，所以需要在springboot内部使用一种机制去通知tomcat容器重启。该机制为：当springboot检查到代码发生变化时，不需要重启整个项目就能加载新的代码
✅热部署的过程之二#
  • 重启（Restart）：自定义开发代码，包含类、页面、配置文件等，加载位置restart类加载器
  • 重载（ReLoad）：jar包，加载位置base类加载器
  • 两者的区别
    • 类加载器不一样，自己手动开发的资源会经常发生变化，而导入jar包通常不会发生变化【除非需要导入新的jar包】，所以热部署的意义在于重新加载restart类加载器，保留了base类加载器
    • 热部署还是重置了内置的服务器容器，也重新进行了编译，只是在大项目编译的过程中节约了时间
✅手动启动热部署步骤#

  • 在pom文件中加入热部署依赖

    • <!--热部署-->
<dependency>
   <groupId>org.springframework.boot</groupId>
   <artifactId>spring-boot-devtools</artifactId>
   <optional>true</optional><!--当项目自身被依赖时，标注依赖是否传递。用于连续依赖时使用-->
</dependency>

  • 使用快捷键【Ctrl +F9】或者点击idea工具栏，就能完成热部署

  • 选项的内容已经移最上方变成蓝色代表了该选项已经生效

  • 当idea失去焦点的5秒后就会自动启动热部署了

  • StatusResultMatchers可以设定各种预期值，来检查虚拟请求是否符合预期【包括状态码，请求头信息等等】

    MockHttpServletRequestBuilder builder = MockMvcRequestBuilders.get("/users");

    //执行请求并返回执行结果
    ResultActions action = mvc.perform(builder);

    //设置预期值类型一：执行状态
    StatusResultMatchers status = MockMvcResultMatchers.status();
    //设置执行后状态为正常，即状态码为200
    ResultMatcher ok = status.isOk();
    //结果与预期值比对
    action.andExpect(ok);

    //响应体
    ContentResultMatchers content = MockMvcResultMatchers.content();
    ResultMatcher result = content.string("user....");
    action.andExpect(result);

    //响应头
    HeaderResultMatchers header = MockMvcResultMatchers.header();
    ResultMatcher resultHeader = header.string("Content-Type", "text/plain;charset=UTF-8");
    action.andExpect(resultHeader);
}

                        .document(book)                     //指定实体对象
                        .id(book.getId().toString())        //指定id

                ).build()                                   
        );
    }

    //bulk()：一个请求中执行多个添加/修改/删除操作
    client.bulk(bulkRequest -> bulkRequest                  //请求类
            .operations(operationList)                      //将集合中的所有元素添加到【请求体】中
    );
}
